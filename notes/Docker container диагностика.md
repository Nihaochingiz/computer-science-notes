## 1. Контейнер не запускается

**Проверка логов и статуса:**

```bash
docker ps -a | grep <имя>                # существует ли контейнер
docker logs <container_id>               # логи приложения
docker inspect <container_id> | grep -i "error\|exit"  # код выхода
```

**Если контейнера нет в `docker ps -a`** — ошибка при создании:

- `docker-compose up` (без `-d`) — посмотреть вывод в реальном времени
- `docker-compose config` — проверить синтаксис yml
- Проверить наличие образа: `docker images`
- Проверить свободное место: `df -h`
- Проверить права доступа к Docker сокету

**Основные причины отказа запуска:**

- Порт занят: `driver failed programming external connectivity`
- Ошибка в Entrypoint: приложение падает сразу после старта
- Отсутствие образа: `Unable to find image locally`
- Нехватка ресурсов: `no space left on device` или OOM

## 2. «No space left on device»

**Диагностика:**

```bash
docker system df          # разбивка по типам: образы, контейнеры, volumes, кэш
df -h                     # общее состояние диска
```

**Что занимает место:**

- Подвешенные образы (dangling images) — слои без тегов
- Остановленные контейнеры — сохраняют файловую систему
- Неиспользуемые volumes
- Build cache — ускоряет сборку, но разрастается

**Очистка:**

```bash
docker system prune                     # остановленные контейнеры, неиспользуемые сети, подвешенные образы
docker system prune -a --volumes        # агрессивная чистка (ВНИМАНИЕ: удаляет named volumes!)
```

**Избирательная чистка:**

```bash
docker container prune      # остановленные контейнеры
docker image prune -a       # неиспользуемые образы
docker volume prune         # неиспользуемые volumes
```

**Профилактика:**

- `docker run --rm` — автоматически удалять контейнер после остановки
- Настроить CI/CD на очистку после сборки
- Для продакшена — вынести `/var/lib/docker` на отдельный раздел

## 3. Проблемы с правами доступа (volumes)

**Симптом:** при монтировании директории приложение в контейнере не может читать/писать файлы.

**Корень:** UID пользователя в контейнере не совпадает с UID владельца файлов на хосте.

**Решение — создать пользователя с нужным UID в Dockerfile:**

```dockerfile
FROM ubuntu:22.04

# Узнать ваш UID на хосте: id -u

RUN groupadd -g 1000 myappuser && \
    useradd -u 1000 -g 1000 -m myappuser

USER myappuser

CMD ["python", "app.py"]
```

**Нерекомендуемые варианты:**

- Запуск от root (опасно для продакшена)
- `chmod 777` на хосте (небезопасно)

## 4. Сетевые проблемы между контейнерами

**Симптом:** контейнер `app` не подключается к `database`, хотя использует `localhost:5432`.

**Корень:** у каждого контейнера свой `localhost`, он не видит другие контейнеры.

**Решение — пользовательская сеть:**

```bash
# Создать сеть
docker network create my_app_network

# Запустить БД в этой сети
docker run -d --name database --network my_app_network postgres:15

# Запустить приложение в этой сети
docker run -d --name app --network my_app_network -p 80:5000 my-app-image
```

**Что это дает:**

- DNS-резолвинг по имени контейнера
- В коде приложения используйте `database:5432` вместо `localhost:5432`
- Изоляция окружений (dev/stage) в разных сетях

## Краткий чек-лист

| Симптом                        | Команда / Решение                                                                                  |
| ------------------------------ | -------------------------------------------------------------------------------------------------- |
| Контейнер не появился          | `docker ps -a`, `docker logs`, проверить `df -h`                                                   |
| Нет места на диске             | `docker system prune -a --volumes` (с осторожностью)                                               |
| Permission denied в volume     | Создать пользователя в Dockerfile с UID = ваш `id -u`                                              |
| Контейнеры не видят друг друга | `docker network create`, запускать в одной сети, использовать имена контейнеров вместо `localhost` |
